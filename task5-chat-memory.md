# Задание 5: мини-чат с RAG + источники + «память задачи» (production-like)

> Самодостаточное задание для сессии в проекте `webchat`. Сначала прочитай `./CLAUDE.md`
> — там инварианты (grounding, границы фронт/backend/RAG, ключ только в env),
> персона-агенты и **гейт Plan Mode**. Это крупная задача — соблюдай гейт.

## Что уже готово (переиспользовать, не переписывать)

Чат-UI, история в `localStorage`, RAG-поиск на каждый вопрос, grounding с
источниками+цитатами и режимом «не знаю», скользящее `summary`. **Новое** —
структурированная «память задачи» + сценарии проверки.

## Решения по scope (согласованы — не пересматривать без причины)

- **Память задачи живёт в BACKEND**, in-memory по `session_id` (dict в процессе;
  durable SQLite — явно «на потом», не сейчас).
- Обновление task-state — **отдельным LLM-вызовом** DeepSeek каждый ход (как
  `summarize`, но structured JSON), не совмещать с генерацией.
- Схема task-state: `{goal, constraints[], terms[], clarified[]}` —
  цель диалога / зафиксированные ограничения / термины / что пользователь уточнил.
- **Мини-чат: RAG всегда включён + источники всегда.**
- Потребить новое поле `chunk_id` из rag `/search` (контракт расширен соседом:
  `chunks[i]` теперь содержит `chunk_id:int`).

## Гейт Plan Mode

Задача меняет архитектуру (новый stateful-слой в backend) → по `CLAUDE.md` **сначала**
запусти в обычном режиме:
- 🏗️ **ARCHITECT** — дизайн слоя памяти (где `session_id`, как хранится, как
  инъектируется в промпт генерации);
- 🧠 **LLM ENGINEER** — промпт извлечения task-state, управление контекстом.

Промпты ≤200 слов, дождись выводов, **потом** `EnterPlanMode`.

## Реализация (после плана)

- **BACKEND** (⚙️ BACKEND DEVELOPER): `session_id` от фронта; in-memory
  `{session_id → task_state}`; на каждый ход LLM-вызов обновляет `task_state` из
  (текущий `task_state` + свежие сообщения); инъекция `task_state` в system-prompt
  генерации рядом с RAG-контекстом; вернуть `task_state` в ответе. Сохранить
  grounding (цитаты/«не знаю») и авто-повтор — **не регрессировать**. Прокинуть
  `chunk_id` в `Source`.
- **FRONTEND** (⚛️ FRONTEND DEVELOPER): генерировать/хранить `session_id` в
  `localStorage` и слать его; панель **«Состояние задачи»** (цель/ограничения/
  термины/уточнённое) из ответа; RAG по умолчанию включён; `chunk_id` в блоке
  источников. Ревью TS/React — 🔍 CODE REVIEWER.
- **QA** (🧪 QA ENGINEER): **2 сценария по 10–15 сообщений** (в `eval/`), прогон через
  backend, assert: `task_state.goal` не теряется по ходу диалога **И** у каждого
  RAG-ответа есть источники. Краткий отчёт.

## Деплой и проверка (операционные факты этой среды — важно)

- **backend**: uvicorn **без** `--reload` → после правок **обязательно**
  `sudo systemctl restart webchat.service` (иначе крутится старый код в памяти).
- **фронт**: `cd frontend && npm run build`, затем скопировать `dist` в
  `/var/www/webchat`. **`rsync` НЕ установлен** — использовать `cp`:
  удалить старые `assets` + `cp -a dist/. /var/www/webchat/`.
- **eval-скрипты** бей напрямую в `http://127.0.0.1:8000` (минуя nginx-гейт/лимит).
- **Финал**: прогнать 2 сценария (цель удержана + источники каждый ход); проверить
  панель памяти в UI (`Ctrl+Shift+R` после редеплоя фронта).
