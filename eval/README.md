# eval/ — сравнение качества RAG vs no-RAG

Реализует «усиление» из задачи: агент с двумя режимами (с RAG / без) + набор из
10 контрольных вопросов по базе знаний и сравнение качества ответов.

## Файлы
- **`questions.json`** — 10 контрольных вопросов. У каждого: `question`,
  `expectation` (что должно быть в ответе), `expected_sources` (какие файлы
  корпуса должны быть использованы).
- **`compare.py`** — гоняет каждый вопрос через backend в **трёх режимах**
  (`no-RAG` · `plain` · `improved`), пишет `results.json` и `REPORT.md`.
- **`results.json`** — сырые ответы всех режимов + источники + переписанный запрос.
- **`REPORT.md`** — сгенерированное сравнение (сводная таблица + тройки ответов).

## Режимы
- **no-RAG** — `useRag:false`, только знания модели.
- **plain** — `useRag:true, improvedRag:false`: одностадийный top-k retrieval.
- **improved** — `useRag:true, improvedRag:true`: query rewrite (DeepSeek) +
  порог-фильтр по similarity + rerank, `k_before`→`k_after`.

## Как запустить
```sh
# backend webchat должен быть поднят (systemd webchat.service),
# а RAG-сервис — на 127.0.0.1:8100 (rag-search.service).
cd eval && python3 compare.py
```
Скрипт бьёт **напрямую в backend `127.0.0.1:8000`**, минуя nginx: так 20 быстрых
запросов не упираются в токен-гейт и rate-limit (`429`). Для прогона через
публичный URL: `BACKEND_URL=https://jorchik.com CHAT_TOKEN=<токен> python3 compare.py`.

## Корпус и почему вопросы код-специфичные
Корпус — Google **compose-samples** (Jetchat, JetNews, Reply, Jetsnack, Jetcaster,
JetLagged): реальные исходники Kotlin + README. Это публичный репозиторий, поэтому
DeepSeek знает сэмплы **на общем уровне** — на вопросах «что такое Jetchat» разницы
почти нет. Ценность RAG видна на **точных деталях кода** (пути файлов, имена
функций, поля классов, конкретные строки): там модель без RAG уверенно выдумывает,
а с RAG отвечает по источнику либо честно говорит, что данных нет.

## Выводы (последний прогон: retrieval hit — plain 9/10, improved 7/10)

**1. RAG резко снижает уверенные галлюцинации** (no-RAG vs plain/improved):
- *Q1 (`addMessage`)*: без RAG — «вставляет в **конец**» + выдуманный код; с RAG —
  верно «в **начало**, `_messages.add(0, msg)`» [источник].
- *Q10 (год копирайта)*: без RAG — «**2022**»; с RAG — «**2024**» [источник].
- *Q2 (конструктор)*: без RAG — выдуман целый конструктор, которого в коде нет.

**2. improved пока НЕ выигрывает у plain по hit-rate — и это ожидаемо.** Причины:
- **Реранкера ещё нет.** Cross-encoder реализуется в `../rag/` (см.
  `../rag-reranker-task.md`). Пока `improved` = rewrite + порог-фильтр над
  **cosine-порядком**; переупорядочивания по релевантности нет.
- **Вопросы самодостаточные.** query rewrite нужен для multi-turn (раскрыть «оно»,
  «этот класс»). На одиночных вопросах он ничего не раскрывает и иногда **уводит**
  запрос → топ-5 по cosine сдвигается, нужный файл выпадает (Q2 и др.).
- Где improved уже полезен: **отсечение off-topic** (порог выкидывает нерелевантный
  шум — на вопросе «как испечь хлеб» фильтр отсекает все 20 чанков, и модель не
  «обосновывает» ответ мусором) и **rewrite в диалоге** (проверяется вручную в UI,
  не этим набором одиночных вопросов).

**3. Качество упирается в retrieval, а не в генерацию.** hit меряется на уровне
**файла**; даже file-level hit не гарантирует, что нужный **чанк** в топ-K (Q5:
файл есть, но чанк с телом `upPress()` не попал). Главный рычаг — retrieval:
формулировка запроса, `k`, стратегия чанкинга и **реранкинг**.

## Как улучшить (следующие шаги)
- **Подключить cross-encoder reranker в `../rag/`** (промт готов: `../rag-reranker-task.md`)
  — тогда improved начнёт поднимать нужный чанк в топ и обгонит plain.
- Мерить hit не только по файлу, но по совпадению ключевых строк ожидания.
- Добавить в набор **multi-turn** вопросы, где query rewrite реально помогает.
- Добавить LLM-судью для авто-оценки соответствия ответа `expectation`.
