# eval/ — сравнение качества RAG vs no-RAG

Реализует «усиление» из задачи: агент с двумя режимами (с RAG / без) + набор из
10 контрольных вопросов по базе знаний и сравнение качества ответов.

## Файлы
- **`questions.json`** — 10 контрольных вопросов. У каждого: `question`,
  `expectation` (что должно быть в ответе), `expected_sources` (какие файлы
  корпуса должны быть использованы).
- **`compare.py`** — гоняет каждый вопрос через backend дважды (`useRag:false` и
  `useRag:true`), пишет `results.json` и `REPORT.md`.
- **`results.json`** — сырые ответы обоих режимов + извлечённые источники.
- **`REPORT.md`** — сгенерированное сравнение (сводная таблица + пары ответов).

## Как запустить
```sh
# backend webchat должен быть поднят (systemd webchat.service),
# а RAG-сервис — на 127.0.0.1:8100 (rag-search.service).
cd eval && python3 compare.py
```
Скрипт бьёт **напрямую в backend `127.0.0.1:8000`**, минуя nginx: так 20 быстрых
запросов не упираются в токен-гейт и rate-limit (`429`). Для прогона через
публичный URL: `BACKEND_URL=https://jorchik.com CHAT_TOKEN=<токен> python3 compare.py`.

## Корпус и почему вопросы код-специфичные
Корпус — Google **compose-samples** (Jetchat, JetNews, Reply, Jetsnack, Jetcaster,
JetLagged): реальные исходники Kotlin + README. Это публичный репозиторий, поэтому
DeepSeek знает сэмплы **на общем уровне** — на вопросах «что такое Jetchat» разницы
почти нет. Ценность RAG видна на **точных деталях кода** (пути файлов, имена
функций, поля классов, конкретные строки): там модель без RAG уверенно выдумывает,
а с RAG отвечает по источнику либо честно говорит, что данных нет.

## Выводы (по последнему прогону: retrieval hit 9/10)

**1. RAG резко снижает уверенные галлюцинации.** Примеры:
- *Q1 (`ConversationUiState.addMessage`)*: без RAG — «вставляет в **конец** списка»
  с выдуманным кодом; с RAG — верно «в **начало**, `_messages.add(0, msg)`» [источник].
- *Q10 (год копирайта в корневом README)*: без RAG — «**2022**»; с RAG — «**2024**» [источник].
- *Q2 (конструктор `ConversationUiState`)*: без RAG — выдуман целый конструктор
  (`profile, reply, input, isLoading…`), которого в коде нет.

**2. Качество RAG упирается в retrieval.** «Ожидаемый источник извлечён?» меряется
на уровне **файла**. Даже это не гарантирует ответ:
- *Q2*: нужный файл не попал в топ-5 → модель честно сказала «в контексте нет»
  (graceful, без выдумки) — но и не ответила. Один и тот же файл при формулировке
  Q1 извлекается, при Q2 — нет: retrieval чувствителен к формулировке.
- *Q5*: файл извлёкся (hit ✅), но нужный **чанк** (с телом `upPress()`) в топ-5 не
  попал → снова «в контексте нет». File-level hit ≠ факт в контексте.

**Итого:** там, где нужный чанк извлечён, RAG даёт точный ответ со ссылкой; без RAG
модель на код-специфике уверенно ошибается. Главный рычаг улучшения — не генерация,
а **retrieval** (формулировка запроса, `k`, стратегия чанкинга, реранкинг).

## Как улучшить (следующие шаги)
- Поднять `k` (сейчас 5 на backend) или добавить реранкер, чтобы нужный чанк реже
  выпадал из окна (см. Q2, Q5).
- Мерить hit не только по файлу, но по совпадению ключевых строк ожидания.
- Добавить LLM-судью для авто-оценки соответствия ответа `expectation`.
