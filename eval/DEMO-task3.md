# Задание 3 — шпаргалка для видео (реранкинг и фильтрация)

Что показываем **прямо в веб-чате** на [jorchik.com](https://jorchik.com): **query
rewrite** раскрывает вопрос по контексту диалога, а **порог-фильтр** отсекает
нерелевантное и честно отвечает «не знаю». В конце — как показать **сравнение
режимов в терминале** (наш воспроизводимый результат).

Задавать нужно только текст вопроса (в блоках `код` ниже).

---

## Подготовка

1. Открыть [jorchik.com](https://jorchik.com), начать новый чат (очистить историю, если есть).
2. Под полем ввода два чекбокса: **«RAG (поиск по базе знаний)»** и вложенный
   **«Улучшенный RAG»** (появляется, только когда включён первый).
3. Во всех трёх сценариях ниже — **оба чекбокса включены** (это и есть «улучшенный
   RAG» из задания), кроме контрольного сценария 03.

---

## Сценарий 01 — query rewrite на многоходовом диалоге

**Доказывает:** во втором вопросе нет ключевых слов, только «**it**». Улучшенный
режим переписывает его в самостоятельный запрос по контексту, поэтому находит файл.

**Чекбоксы:** ☑ RAG · ☑ Улучшенный RAG

**Задать по очереди** (второе — дождавшись ответа на первое):

```
Tell me about the Reply sample's navigation.
```
```
How does it decide which one to show?
```

**Ждём:** ответ по существу со ссылкой на `Reply/…/ReplyNavigationComponents.kt`
(выбор навигации по размеру окна) — хотя в вопросе было только «it».

---

## Сценарий 02 — порог-фильтр → режим «не знаю»

**Доказывает:** реранкер даёт нерелевантным чанкам резко низкий скор (в теме ≈ +2.7,
вне темы ≈ −11); всё ниже порога −6 отсекается → модель не выдумывает, а воздерживается.

**Чекбоксы:** ☑ RAG · ☑ Улучшенный RAG

**Задать вопрос не по базе:**

```
What is the capital of Australia?
```

**Ждём:** «**Не знаю: в найденных источниках нет ответа на этот вопрос. Уточните или
переформулируйте вопрос.**» Блок «Источники» пустой — хотя ответ модель знает.

---

## Сценарий 03 — контроль: без RAG «не знаю» не срабатывает

**Доказывает контраст (сравнение режимов):** тот же off-topic **без RAG** модель
охотно отвечает из своих знаний. Значит воздержание в 02 — заслуга фильтра.

**Чекбоксы:** ☐ RAG (оба выключены)

**Тот же вопрос:**

```
What is the capital of Australia?
```

**Ждём:** обычный ответ `Canberra`, без источников.

## Числа реранка в чате не видны — это нормально

Веб-чат показывает **эффекты** реранка (точные ответы, отсечение off-topic), но не
сами скоры и топ-K. Чтобы показать саму механику с цифрами — см. терминальную секцию.

---

## Показать сравнение в терминале (наш воспроизводимый результат)

Всё на VPS. Backend (`webchat.service`) и RAG-сервис (`rag-search.service`) должны быть подняты.

### 1. Сравнение трёх режимов: no-RAG · plain · improved

```sh
cd ~/repos/webchat/eval && python3 compare_rag.py
```

Гоняет 12 контрольных вопросов в трёх режимах и печатает сводку, а также пишет
`eval/REPORT-modes.md` (таблица + ответ каждого режима по каждому вопросу). Ожидаемый
итог:

```
retrieval hit — plain 9/12 · improved 12/12 (отрыв +3)
```

Что проговорить:
- **Задание 2 (no-RAG vs RAG):** в `REPORT-modes.md` видно, как без RAG модель
  отвечает из общих знаний (и на точных деталях кода ошибается), а с RAG — по
  источнику либо «не знаю».
- **Задание 3 (plain vs improved):** improved обгоняет plain на **+3** — прирост
  дают multi-turn вопросы (11, 12), где срабатывает query rewrite, и вопрос 2,
  где нужный чанк поднимает реранкер.

### 2. Механика реранка с числами (из репозитория `../rag`)

```sh
cd ~/repos/rag && bash demo_task3.sh
```

Показывает на `/search` напрямую: без реранка нужный чанк тонет по косинусу; с
реранком cross-encoder поднимает его на 1-е место (`+2.75`); off-topic получает
скор ≈ −11, что ниже порога −6 → отсекается.
