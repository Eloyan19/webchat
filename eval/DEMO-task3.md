# Задание 3 — реранкинг и фильтрация: демонстрация

Демонстрация улучшенного RAG в веб-чате [jorchik.com](https://jorchik.com): query
rewrite и порог-фильтр. В конце — воспроизводимое сравнение режимов в терминале.

## Подготовка

- Открыть [jorchik.com](https://jorchik.com), очистить историю чата.
- Под полем ввода два чекбокса: **«RAG (поиск по базе знаний)»** и вложенный
  **«Улучшенный RAG»** (виден только при включённом RAG).

---

## 1. Query rewrite (многоходовой диалог)

**Опции:** ☑ RAG · ☑ Улучшенный RAG

**Задать по очереди** (второе — после ответа на первое):

```
Tell me about the Reply sample's navigation.
```
```
How does it decide which one to show?
```

**Результат:** ответ со ссылкой на `Reply/…/ReplyNavigationComponents.kt` (выбор
навигации по размеру окна).

**Почему:** второй вопрос содержит только «it», без ключевых слов. Улучшенный режим
сперва переписывает его в самостоятельный запрос по контексту диалога, и уже он идёт
в retrieval. Без rewrite (plain) этот вопрос как поисковый запрос не находит файл.

---

## 2. Порог-фильтр → «не знаю»

**Опции:** ☑ RAG · ☑ Улучшенный RAG

**Вопрос не по базе:**

```
What is the capital of Australia?
```

**Результат:** «Не знаю: в найденных источниках нет ответа на этот вопрос. Уточните
или переформулируйте вопрос.» Блок «Источники» пустой.

**Почему:** cross-encoder даёт нерелевантным чанкам резко низкий `rerank_score`
(в теме ≈ +2.7, вне темы ≈ −11). Всё ниже порога `RERANK_THRESHOLD = -6.0`
отсекается; когда не остаётся ни одного чанка — backend отвечает заранее заданным
«не знаю», без вызова генерации.

---

## 3. Контроль: без RAG порог не участвует

**Опции:** ☐ RAG (оба выключены)

**Тот же вопрос:**

```
What is the capital of Australia?
```

**Результат:** обычный ответ `Canberra`, без источников.

**Почему:** при `useRag=false` retrieval и фильтр не задействованы — модель отвечает
из своих знаний. Разница с п.2 показывает, что воздержание там — работа фильтра.

---

## Сравнение режимов в терминале

Backend (`webchat.service`) и RAG-сервис (`rag-search.service`) должны быть подняты.

### no-RAG · plain · improved

```sh
cd ~/repos/webchat/eval && python3 compare_rag.py
```

**Результат:** сводка в консоли + `eval/REPORT-modes.md` (таблица + ответ каждого
режима по каждому вопросу):

```
retrieval hit — plain 9/12 · improved 12/12 (отрыв +3)
```

**Почему +3:** вопросы 11 и 12 — multi-turn, их берёт query rewrite; вопрос 2 —
нужный чанк поднимает реранкер. Метрика (retrieval hit по файлу) детерминирована;
текст ответов между прогонами может отличаться.

### Механика реранка с числами (репозиторий `../rag`)

```sh
cd ~/repos/rag && bash demo_task3.sh
```

**Результат:** на `/search` напрямую — без реранка нужный чанк тонет по косинусу;
с реранком cross-encoder поднимает его на 1-е место (`+2.75`); off-topic получает
`rerank_score` ≈ −11, ниже порога −6.
