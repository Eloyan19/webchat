## Терминал: реранк, порог, топ-K

Backend (`webchat.service`) и RAG-сервис (`rag-search.service`) должны быть подняты.

### Механика реранка и порога

```sh
cd ~/repos/rag && bash demo_task3.sh
```

**Результат (три шага):**
1. **без реранка** — только косинус: нужный чанк `ConversationUiState` не попадает в топ;
2. **с реранком** — cross-encoder поднимает его на 1-е место (`rerank_score ≈ +2.75`);
3. **off-topic** — все чанки получают `rerank_score` ≈ −11, ниже порога `−6` → отсекаются
   (топ-K 20 → 0).

**Почему:** косинус nomic «сжат» (в теме ~0.66, off-topic ~0.50) — одного порога мало.
Cross-encoder оценивает пару (запрос, чанк) совместно и даёт резкое разделение
relevant/off-topic, поэтому порог `−6` надёжно отсекает нерелевантное.

### Сравнение режимов: no-RAG · plain · improved

```sh
cd ~/repos/webchat/eval && python3 compare_rag.py
```

**Результат:** сводка + `eval/REPORT-modes.md`. Improved стабильно обгоняет plain;
типичный прогон — `plain 7/12 · improved 10/12`.

**Почему improved > plain:** multi-turn вопросы (11, 12) берёт query rewrite; на части
одиночных нужный чанк поднимает реранкер (напр. вопрос 2). Точные счётчики плавают
(цитату и rewrite генерирует LLM); стабильно — improved > plain.
